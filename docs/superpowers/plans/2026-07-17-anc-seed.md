## What ANC is

ANC is a **CREATE** feed (message_kind `create`, per the seeder design doc §"Per-feed shape").
Unlike FD, an ANC message does not represent a disruption — it represents an **already-purchased
ancillary** (a paid seat selection or extra bag). The seed therefore has to make the ancillary
*exist and be purchased* in the target system before the chat can reason about refunding it.

Case count (parsed by `catalog.parser.load_catalog`): **54 cases**, all `seed_pending`
(the gap doc carries no embedded PNR dataset / datagrid), all titled. Flows split into `seat` and
`bag` variants (`data-flow="seat" | "bag"`).

## Seed mechanism — EMD-based CREATE

Where FD publishes `pnr + ticket + fdm(skd/delay)`, ANC publishes:

1. **PNR** — same base PNR create as FD (traveler names, journey/segments, contact).
2. **Ticket** — the fare ticket (same shape as FD's `02_ticket_issue`).
3. **EMD** — the **Electronic Miscellaneous Document** issue message (`03_emd_issue.json` in the
   reference), **plus its correlation message** tying the EMD to the ticket/PNR. The EMD is what FD
   does not have: it is the record that a *paid ancillary* (seat fee or bag fee) was issued against
   the booking. Fields the EMD adds beyond the FD trio:
   - EMD document number (the `014`-prefixed AC document; some cases deliberately use a non-`014`
     document to force an OAL/other-airline referral — see `ANC-SEAT-TC-003`).
   - EMD **status** — `I` issued / `R` refunded / `V` voided / `E` exchanged. Several cases pin an
     already-`R` (refunded) or `V` (voided) EMD to test the "already handled" branch
     (`SEAT_TC001` = *EMD Already Refunded (Status R)*, `ANC-SEAT-TC-002` = *EMD Voided (Status V)*).
   - The service subcode / RFISC identifying seat-selection vs baggage.
   - The **correlation** record linking the EMD to the coupon/segment so the refund-eligibility
     detector can find it.

## Refund DDS path

Refund-eligibility variants additionally **pin a DDS** (design doc: `dds: seat_refund | bag_refund`).
The DDS array the verdict fills depends on the flow:

- **seat** flow → `seatFeeRefundEligibility[]`
- **bag** flow → `baggageRefundEligibility[]`

Each eligibility entry carries the per-EMD refund verdict (eligible / not-eligible / amount /
reason). This mirrors FD's `compensation` DDS and SOC's `socFlightEligibility` — same pin mechanism,
different array name. Non-refund cases (pure "already refunded/voided" status checks) need **no DDS**
— the EMD status alone drives the answer.

The `anc.yaml` judge is `match_on: [status, system_code]` where `status` = the refund verdict and
`system_code` = the EMD status, so the deterministic judge can assert both the refund outcome and the
underlying EMD state.

## How it uses the engine

ANC fits the generic seed engine (design doc §Architecture) with **no prelude** (it is a pure CREATE,
not an UPDATE — there is no change-event, so no CREATE-prelude is required):

1. evaluate identity — fresh env-specific PNR + today-relative date from the scenario's temporal
   intent (ANC refund cases are typically `completed`, i.e. `today-7d`, since the ancillary was
   already flown/consumed).
2. pick base template + variation — `data/seed-templates/anc/` with `base.json` (pnr+tkt+emd) and
   variations keyed by EMD status (`I`/`R`/`V`) and flow (`seat`/`bag`).
3. rekey the id-family — `pnr_id`, `PT-n`, `ST-n`, ticket/EMD document numbers.
4. apply manifest mutations — set EMD status, RFISC/subcode, document prefix (`014` vs OAL).
5. gate — test-safe locator/document ranges.
6. prelude — **none**.
7. publish the PNR + ticket + EMD + correlation as one burst; settle.
8. pin DDS — `seatFeeRefundEligibility[]` or `baggageRefundEligibility[]` from the scenario verdict
   (refund variants only).
9. verify checkpoints — `anc` auditor: `emd_issued`, `emd_correlation`,
   `dds_seat_refund_eligibility`, `dds_bag_refund_eligibility`, plus the shared trip/passenger/EDS
   areas.

## Build steps

- [ ] Add `data/seed-templates/anc/{base.json, variations/, manifest.yaml}` (pnr+tkt+emd+corr).
- [ ] Manifest identity: fresh PNR + `scenario_date(completed)`; mutable: EMD status, RFISC, doc prefix.
- [ ] EMD golden: one rendered EMD-issue + correlation validates against schema (offline).
- [ ] DDS builder: verdict → `seatFeeRefundEligibility[]` / `baggageRefundEligibility[]`.
- [ ] `anc` auditor checkpoints (live; built when ANC is onboarded to a live env).

## Open items / grounding notes

- The gap doc has **no tabular dataset** — the 54 cases are `seed_pending`. A real ANC run needs
  either an EMD-aware dataset joined via `catalog.parser.join_dataset`, or the generic-seeder
  manifest to synthesise the EMD family per case. The registry `columns` map placeholder column names
  (`EMD status`, `Refund amount`, …) so a future dataset can join without a schema change.
- `system_code` parses as `—` (the gap-doc `badge req` placeholder) and `verdict` is empty (the doc
  uses `data-flow`, not `data-out`). This is a property of how the ANC doc was authored, **not** a
  parse failure; the EMD status / refund verdict live in the case titles for now.
